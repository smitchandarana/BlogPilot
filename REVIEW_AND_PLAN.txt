============================
KNOWN REMAINING RISKS
============================

1. LinkedIn DOM selectors (Medium severity)
   LinkedIn's frontend updates without notice. The feed scanner uses 7 fallback selectors
   for author name, which reduces brittleness, but any major LinkedIn DOM overhaul will
   break extraction silently (posts will be found but fields will be null/empty).
   Mitigation: run the engine periodically, monitor the Feed Engagement page for null
   author names. When selectors break, update backend/automation/feed_scanner.py and
   backend/automation/profile_scraper.py. No structural code change needed.

2. Groq TPD (Tokens Per Day) limit (Medium severity)
   Groq's free tier has a daily token cap (~100k tokens/day). The Content Intelligence
   extraction job runs every 4 hours and processes up to 20 snippets per batch. If the
   research pipeline is active and bulk extraction is triggered, the daily cap can be hit
   before comment generation runs. This causes generation tasks to fail with a 429 that
   is logged but not surfaced to the user prominently.
   Mitigation: the client_factory routes background tasks (extraction, relevance scoring)
   to OpenRouter (free tier, large context) and reserves Groq for generation tasks. Add
   an OpenRouter API key in Settings to enable this split. Without an OpenRouter key,
   all tasks share Groq's TPD.

3. LinkedIn detection risk (Medium severity)
   The automation acts on LinkedIn's platform. The circuit breaker, rate limiter, human
   behavior delays, and worker pool cap (3 threads) all reduce detection risk, but they
   do not eliminate it. Operating outside the configured activity window (start_hour to
   end_hour), or running with aggressive daily limits, raises risk. The tool is designed
   for managed, human-supervised use — not fully autonomous long-run operation.

4. Cookie expiry and re-authentication (Low severity)
   LinkedIn session cookies expire. The login module detects re-auth redirects and
   calls handle_security_challenge(), which pauses the engine for 120 seconds and
   broadcasts an alert. The user must manually complete re-authentication in the browser
   window. There is no automated recovery path (by design — automating re-auth is a
   detection risk).

5. Profile scraper field completeness (Low severity)
   The profile scraper uses expanded fallback selectors but LinkedIn's profile page
   structure varies by account type, connection degree, and A/B tests LinkedIn runs.
   Fields (title, company) may occasionally return null for some profiles.
   This does not break the pipeline — it creates leads with partial data that the user
   can manually enrich.

6. Windows-only lock file (Low severity, known)
   backend/utils/lock_file.py uses msvcrt (Windows-specific). The code works correctly
   on Windows. If the app is ever run on Linux/macOS, the lock file will need a
   fcntl-based implementation. Not a concern for Phase 1 (Windows desktop app).

7. No API rate limiting middleware (Low severity)
   The FastAPI server has no rate limiting on API endpoints. The /prompts/test endpoint
   in particular calls Groq on every request. In local single-user mode this is not a
   problem. If the backend is ever exposed on a network, add rate limiting middleware
   before doing so.

PHASE 2 KICKOFF CRITERIA
============================

M7 is considered validated when ALL of the following are true after a 4-hour engine run:

  1. scripts/m7_validate.py exits 0 (no failures, warnings acceptable)
  2. Budget counters for at least 2 action types show non-zero count_today values
  3. At least 10 posts appear in the database with a non-SEEN state (ACTED or SKIPPED)
  4. The engine did not crash (process still running at end of run)
  5. The midnight budget reset ran correctly (verified by check 4 in m7_validate.py)
  6. No CAPTCHA or re-auth events in this run (clean session)
